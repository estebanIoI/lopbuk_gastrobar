### ⚠️ Antes de desplegar (crítico)
El fix **F2 es fail-closed**: si tu `.env` de **producción** no tiene `JWT_SECRET` (≥32 chars) y
`ENCRYPTION_KEY` (≥64 hex), **el backend abortará el arranque**. Esto es intencional (mejor caído que
inseguro), pero **confírmalo antes del deploy**:
```bash
# en el .env de producción deben existir y ser largos/aleatorios:
JWT_SECRET=...>=32 chars...
ENCRYPTION_KEY=...64 hex chars (32 bytes)...
```
Como la PII ya se cifra en prod, `ENCRYPTION_KEY` seguramente ya está seteada; verifica `JWT_SECRET`.

### Verificación sugerida
```bash
cd backend && npx tsc --noEmit        # o tu script de check
# Prueba manual del cierre de F1 (en entorno controlado): debe dar 400/403
curl -X POST http://TU_API/api/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"email":"x@x.com","password":"123456","name":"x","role":"superadmin"}'
```

## 🟢 Lo que ya estaba bien en identidad/acceso (confirmado)

- `authenticate` revalida `is_active`, `can_login` y estado del tenant en **cada** request (revocación real).
- `authorize(...roles)` y `requirePermission(...)` correctos; `tenant_id` siempre del JWT.
- Endpoints de direcciones (`/auth/addresses/:id`) verifican **ownership** (`WHERE id = ? AND user_id = ?`) → sin IDOR.
- `updateProfile` usa whitelist de campos (no hay mass assignment de `role`/`tenant`).
- Lockout de login + `authLimiter` por IP + auditoría de login/logout.
